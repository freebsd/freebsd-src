## 6.1 Executive Summary

This document defines a lightweight, built-in orchestration layer for FreeBSD native containers that provides Kubernetes-like functionality without the complexity and overhead of Kubernetes itself. The system is designed to be:

- **Simple:** Configuration is declarative and concise, avoiding the verbosity of Kubernetes YAML
- **Integrated:** Networking (Flannel/MetalLB equivalents), storage, and failover are built-in, not bolted-on
- **Resilient:** Automatic failover, health checking, and self-healing are core features
- **Efficient:** Leverages FreeBSD's native jail, ZFS, and VNET capabilities directly

**Key Principle:** A Kubernetes config that spans 200+ lines should be expressible in our format in under 50 lines, while maintaining the same semantic power.

### 6.2.2 Core Concepts

| Concept | Kubernetes Equivalent | FreeBSD Native Implementation |
|---------|----------------------|------------------------------|
| **Pod** | Pod | A group of jails sharing VNET, namespace, and storage |
| **Service** | Service | Virtual IP + port proxy with round-robin load balancing |
| **Stack** | Deployment/StatefulSet | Declarative multi-pod application definition |
| **Volume** | PersistentVolume | ZFS dataset with snapshot/send/recv replication |
| **Network** | NetworkPolicy + CNI | VNET + bridge + pf rules (built-in) |
| **Ingress** | Ingress | pf + haproxy/nginx built-in proxy |
| **Namespace** | Namespace | Jail name prefix + RCTL rules + MAC labels |
| **ConfigMap** | ConfigMap | Nullfs-mounted files or jail environment variables |
| **Secret** | Secret | ZFS encrypted dataset or MAC-protected files |

## 6.3 Configuration Format

### 6.3.1 Design Principles

1. **Flat over nested:** Prefer top-level keys over deep nesting
2. **Sensible defaults:** Most fields are optional; defaults work out of the box
3. **Native types:** Use FreeBSD-native concepts (jail, ZFS, VNET) directly
4. **One file per stack:** A complete application fits in one readable file
5. **JSON-compatible:** YAML is primary, but JSON works identically

### 6.3.2 Stack Definition Example

```yaml
# myapp.yml — FreeBSD Native Container Stack
stack: myapp
version: "1.0"

# Global defaults for all pods in this stack
defaults:
  image: freebsd:latest
  network: myapp-net
  namespace: myapp
  restart: always
  resources:
    memory: 512M
    cpus: 1.0
    pids: 100

# Networks are defined inline, not in separate files
networks:
  myapp-net:
    driver: bridge
    subnet: 192.168.100.0/24
    gateway: 192.168.100.1
    # Built-in MetalLB equivalent — no external dependency
    external_ips:
      - 10.0.0.50/32
    # Built-in Flannel equivalent — automatic overlay
    vxlan: true

# Volumes use ZFS natively
volumes:
  myapp-data:
    driver: zfs
    dataset: zroot/myapp/data
    size: 10G
    # Automatic replication to other hosts
    replicate:
      targets:
        - host: node2.example.com
          dataset: zroot/myapp/data
      interval: 5m

# Pods define the actual containers
pods:
  web:
    replicas: 3
    image: myregistry/nginx-freebsd:1.25
    ports:
      - "80:8080"      # host:container
      - "443:8443"
    volumes:
      - myapp-data:/var/www/html
    environment:
      NGINX_WORKERS: "4"
    healthcheck:
      cmd: ["/usr/local/bin/curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 5s
      retries: 3
    resources:
      memory: 256M
      cpus: 0.5

  api:
    replicas: 2
    image: myregistry/myapp-api:v2.1.0
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: "postgres://db:5432/myapp"
      REDIS_URL: "redis://cache:6379"
    healthcheck:
      cmd: ["/usr/local/bin/curl", "-f", "http://localhost:3000/health"]
      interval: 10s
    resources:
      memory: 512M
      cpus: 1.0

  db:
    replicas: 1
    image: postgres:15-freebsd
    ports:
      - "5432:5432"
    volumes:
      - myapp-data:/var/db/postgres
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: myapp
    # Stateful pod — gets stable hostname and storage
    stateful: true
    resources:
      memory: 1G
      cpus: 1.0

  cache:
    replicas: 1
    image: redis:7-freebsd
    ports:
      - "6379:6379"
    resources:
      memory: 256M

# Services define how pods are exposed
services:
  web-lb:
    type: loadbalancer
    selector:
      pod: web
    ports:
      - port: 80
        target_port: 8080
      - port: 443
        target_port: 8443
    # Built-in — no MetalLB installation needed
    external_ips:
      - 10.0.0.50

  api-lb:
    type: clusterip
    selector:
      pod: api
    ports:
      - port: 3000
        target_port: 3000

# Ingress rules (built-in — no IngressController needed)
ingress:
  - host: myapp.example.com
    service: web-lb
    port: 80
  - host: api.example.com
    service: api-lb
    port: 3000
    tls:
      cert: /etc/ocifbsd/certs/api.crt
      key: /etc/ocifbsd/certs/api.key
```

### 6.3.3 Comparison: Kubernetes vs. Native Format

**Kubernetes (200+ lines):**
```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: myapp
---
apiVersion: v1
kind: ConfigMap
metadata:
  name: myapp-config
  namespace: myapp
data:
  NGINX_WORKERS: "4"
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  namespace: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
        resources:
          requests:
            memory: "256Mi"
            cpu: "500m"
        livenessProbe:
          httpGet:
            path: /health
            port: 80
          initialDelaySeconds: 30
          periodSeconds: 10
---
apiVersion: v1
kind: Service
metadata:
  name: web-service
  namespace: myapp
spec:
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 80
  type: LoadBalancer
---
# ... and more for ingress, volumes, secrets, etc.
```

**Native Format (50 lines):**
```yaml
stack: myapp
version: "1.0"

pods:
  web:
    replicas: 3
    image: nginx:1.25
    ports: ["80:80"]
    resources:
      memory: 256M
      cpus: 0.5
    healthcheck:
      cmd: ["curl", "-f", "http://localhost:80/health"]
      interval: 30s

services:
  web-lb:
    type: loadbalancer
    selector: { pod: web }
    ports: [{ port: 80, target_port: 80 }]
```

## 6.4 Built-in Networking

### 6.4.1 VNET + Bridge (Default)

Every pod gets its own VNET jail with a virtual Ethernet pair:

```
Host:    epair0a ── bridge0 ── epair1a ── epair2a
              │                    │            │
Jail 1:  epair0b (192.168.100.10)            │
Jail 2:                      epair1b (192.168.100.11)
Jail 3:                                   epair2b (192.168.100.12)
```

### 6.4.2 Built-in Flannel Equivalent (VXLAN Overlay)

For multi-host networking, VXLAN is built-in — no separate Flannel installation:

```yaml
networks:
  myapp-net:
    driver: bridge
    subnet: 192.168.100.0/24
    vxlan: true           # Enable cross-host overlay
    vxlan_id: 100
    vxlan_port: 4789
    vxlan_group: 239.1.1.1
```

This creates:
- `vxlan100` interface on each host
- Automatic VXLAN endpoint discovery via multicast or static config
- Pod IPs are routable across all hosts in the cluster

### 6.4.3 Built-in MetalLB Equivalent (Load Balancer)

External IPs are assigned directly by the orchestrator — no separate MetalLB:

```yaml
services:
  web-lb:
    type: loadbalancer
    selector: { pod: web }
    ports: [{ port: 80 }]
    external_ips:
      - 10.0.0.50        # Announced via ARP/NDP or BGP
      - 10.0.0.51
```

Modes:
- **ARP/NDP:** Layer 2 — responds to ARP/Neighbor Discovery for the IP
- **BGP:** Layer 3 — announces via BGP to upstream routers (using `bgpd` or built-in)
- **Static:** IP is assigned to a host interface directly

### 6.4.4 Service Discovery (Built-in DNS)

Every stack gets automatic DNS entries:

```
web.myapp.svc.cluster.local    → 192.168.100.10, 192.168.100.11, 192.168.100.12
api.myapp.svc.cluster.local    → 192.168.100.20, 192.168.100.21
db.myapp.svc.cluster.local     → 192.168.100.30
```

Implemented via:
- `unbound` or `nsd` running on the host
- Automatic zone file generation from stack definitions
- SRV records for port discovery

### 6.4.5 Round-Robin Load Balancing

The orchestrator maintains a connection table and distributes traffic:

```
Client ──► 10.0.0.50:80 ──► [web-1, web-2, web-3] (round-robin)
                │
                └──► Tracks active connections per backend
                └──► Health checks remove failed backends
                └──► Sticky sessions optional (cookie-based)
```

Implementation options:
1. **pf + relayd:** Native FreeBSD tools
2. **nginx/haproxy jail:** Lightweight proxy container
3. **Built-in proxy:** Go/Rust proxy in the orchestrator daemon

## 6.5 Shared Volumes and Storage

### 6.5.1 ZFS-Native Volumes

Volumes are ZFS datasets with automatic snapshot and replication:

```yaml
volumes:
  myapp-data:
    driver: zfs
    dataset: zroot/myapp/data
    size: 10G
    # Automatic snapshots
    snapshot:
      schedule: "0 * * * *"   # Hourly
      retention: 24            # Keep 24 snapshots
    # Cross-host replication
    replicate:
      targets:
        - host: node2.example.com
          dataset: zroot/myapp/data
      interval: 5m
```

### 6.5.2 Volume Propagation

When a pod is rescheduled to another host:

1. **ZFS send/recv:** Latest snapshot is sent to the new host
2. **Incremental sync:** Only changed blocks are transferred
3. **Mount:** Dataset is mounted in the new jail
4. **Failover time:** Typically < 5 seconds for small datasets

### 6.5.3 Read-Only and Read-Write Modes

```yaml
volumes:
  - myapp-data:/var/www/html:ro      # Read-only
  - myapp-data:/var/www/uploads:rw   # Read-write (default)
```

### 6.5.4 Temporary Volumes

```yaml
volumes:
  - type: tmpfs
    target: /tmp
    size: 100M
```

## 6.6 Failover and Resilience

### 6.6.1 Health Checking

Three types of health checks:

1. **Liveness:** Is the pod running? (default: process exists)
2. **Readiness:** Is the pod ready to serve traffic? (custom command or HTTP)
3. **Startup:** Has the pod finished starting? (custom command, disables liveness until done)

```yaml
pods:
  web:
    healthcheck:
      cmd: ["/usr/local/bin/curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 60s   # Don't fail during startup
```

### 6.6.2 Automatic Failover

When a pod fails health checks:

1. **Remove from service:** Stop sending traffic to the failed pod
2. **Restart:** Attempt to restart the jail (up to `max_restarts`)
3. **Reschedule:** If restart fails, start a new instance on another host
4. **Volume migration:** Replicate ZFS dataset to the new host
5. **Update DNS:** Service discovery points to the new instance

```yaml
pods:
  web:
    replicas: 3
    restart: always
    max_restarts: 5
    restart_window: 120s   # Reset restart count after 2 minutes
    # Reschedule to another host if this host fails
    reschedule:
      on_host_failure: true
      on_health_check_failure: true
      preferred_hosts: [node1, node2, node3]
```

### 6.6.3 Rolling Updates

Update pods one at a time to maintain availability:

```bash
# Update image version
ocifbsd stack update myapp --image myapp:v2.0.0

# Rolling update with custom parameters
ocifbsd stack update myapp \
  --image myapp:v2.0.0 \
  --rolling \
  --parallelism 1 \
  --delay 30s
```

Process:
1. Start new pod with new image
2. Wait for health check to pass
3. Add new pod to service
4. Remove old pod from service
5. Stop old pod
6. Repeat for next pod

### 6.6.4 Rollback

```bash
# Rollback to previous version
ocifbsd stack rollback myapp

# Rollback to specific version
ocifbsd stack rollback myapp --revision 3
```

## 6.7 Namespaces and Isolation

### 6.7.1 Stack Namespaces

Each stack runs in its own namespace:

```yaml
stack: myapp
namespace: production
```

Isolation provided:
- **Network:** Separate VNET or VLAN per namespace
- **Process:** Jail name prefix (`production-myapp-web-1`)
- **Resource:** RCTL rules scoped to namespace
- **MAC:** MAC labels per namespace (`mac_biba:high`)
- **Storage:** ZFS dataset per namespace (`zroot/pods/production/myapp`)

### 6.7.2 Multi-Tenancy

```yaml
# tenant-a.yml
stack: api
namespace: tenant-a
network_policy:
  ingress:
    - from:
        - namespace: tenant-a
  egress:
    - to:
        - namespace: tenant-a
        - external: ["8.8.8.8/32"]   # DNS only
```

## 6.8 Image Distribution

### 6.8.1 Local Registry

The orchestrator includes a lightweight local registry:

```yaml
# Built-in registry on each host
registry:
  enabled: true
  listen: "localhost:5000"
  storage: "zfs:zroot/registry"
```

### 6.8.2 Automatic Image Distribution

When a pod is scheduled to a host:

1. Check if image exists locally
2. If not, pull from:
   - Local registry (fastest)
   - Upstream registry (Docker Hub, private registry)
   - Another host in the cluster (peer-to-peer)
3. Verify image signature (optional)
4. Cache in ZFS dataset

### 6.8.3 Image Caching

```yaml
# Pre-pull images on all hosts
ocifbsd image cache myregistry/myapp:v2.0.0

# Cache is automatically managed
# Least-recently-used eviction when storage is full
```

```bash
# Deploy a stack
ocifbsd stack deploy -f myapp.yml

# List stacks
ocifbsd stack ls

# Inspect a stack
ocifbsd stack inspect myapp

ocifbsd stack update myapp --image myapp:v2.0.0
```

## 6.12 Conclusion

The Jail Orchestration System provides Kubernetes-like functionality with:

- **1/4 the configuration complexity:** 50 lines vs. 200+ lines for equivalent deployments
- **Zero external dependencies:** Networking, load balancing, and storage are built-in
- **Native FreeBSD performance:** Direct jail, ZFS, and VNET integration
- **Automatic resilience:** Health checking, failover, and rolling updates out of the box

This system eliminates the need to run Kubernetes on FreeBSD while providing the same core orchestration capabilities.
