## 8.1 Executive Summary

This document defines the namespace isolation, resource monitoring, health monitoring, and live migration capabilities of the FreeBSD native container orchestration system. These features ensure that workloads run reliably, efficiently, and with minimal disruption during maintenance or resource pressure events.

**Key Principles:**
- **Namespaces provide true isolation** — not just naming prefixes, but resource boundaries, network policies, and security domains
- **Health monitoring is proactive** — detect issues before they cause failures
- **Resource monitoring drives decisions** — automatic migration based on actual load, not static limits
- **Live migration is graceful** — drain connections, replicate state, switch over with minimal disruption
- **Load balancing is continuous** — redistribute pods across hosts as conditions change

## 8.2 Namespaces

### 8.2.1 Namespace Isolation Model

Namespaces in the native orchestrator provide multi-tenant isolation without the complexity of Kubernetes namespaces. A namespace is a logical boundary that groups stacks, pods, networks, and volumes.

```
┌─────────────────────────────────────────────────────────────┐
│                      Host System                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  Namespace: production                               │    │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐               │    │
│  │  │ Stack A │ │ Stack B │ │ Stack C │               │    │
│  │  │ (web)   │ │ (api)   │ │ (db)    │               │    │
│  │  └─────────┘ └─────────┘ └─────────┘               │    │
│  │  Network: prod-net (192.168.10.0/24)                │    │
│  │  RCTL: memory.max=32G, cpu.pct=80%                  │    │
│  │  MAC: label prod/...                                │    │
│  └─────────────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  Namespace: staging                                │    │
│  │  ┌─────────┐ ┌─────────┐                           │    │
│  │  │ Stack D │ │ Stack E │                           │    │
│  │  │ (web)   │ │ (api)   │                           │    │
│  │  └─────────┘ └─────────┘                           │    │
│  │  Network: staging-net (192.168.20.0/24)           │    │
│  │  RCTL: memory.max=8G, cpu.pct=40%                 │    │
│  │  MAC: label staging/...                             │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

### 8.2.2 Namespace Configuration

```yaml
namespace: production

# Namespace-level resource quotas
resources:
  memory:
    limit: 32G
    reservation: 16G
  cpus:
    limit: 16.0
    reservation: 8.0
  pids:
    limit: 10000
  storage:
    limit: 500G

# Namespace-level network policies
network_policy:
  ingress:
    - from:
        - namespace: monitoring
      ports:
        - port: 9100
  egress:
    - to:
        - namespace: database
      ports:
        - port: 5432
    - to:
        - namespace: cache
      ports:
        - port: 6379
    - to:
        - cidr: 0.0.0.0/0
      ports:
        - port: 443

# Namespace-level security
security:
  mac_label: prod/
  securelevel: 2
  allow_privileged: false

# Namespace-level defaults for all stacks
defaults:
  restart: always
  network: prod-net
  resources:
    memory: 512M
    cpus: 1.0
```

### 8.2.3 Namespace Isolation Mechanisms

| Mechanism | Implementation | Purpose |
|-----------|---------------|---------|
| **Jail name prefix** | `namespace.stack.pod` | Naming isolation |
| **VNET separation** | Per-namespace bridge | Network isolation |
| **RCTL rules** | `namespace:*` rules | Resource boundary |
| **MAC labels** | `namespace/` prefix | Security isolation |
| **ZFS datasets** | `zroot/namespace/...` | Storage isolation |
| **pf anchors** | `namespace/*` | Firewall isolation |

### 8.2.4 Cross-Namespace Communication

By default, namespaces are isolated. Explicit policies allow controlled communication:

```yaml
# In namespace: production
network_policy:
  ingress:
    - from:
        - namespace: monitoring
        - namespace: logging
      ports:
        - port: 9100   # node-exporter
        - port: 8080   # app metrics
  egress:
    - to:
        - namespace: database
      ports:
        - port: 5432
```

### 8.2.5 Namespace CLI

```bash
# Create a namespace
ocifbsd namespace create production \
  --memory-limit 32G \
  --cpu-limit 16.0 \
  --mac-label prod/

# List namespaces
ocifbsd namespace ls

# Inspect a namespace
ocifbsd namespace inspect production

# Show namespace resource usage
ocifbsd namespace stats production

# Delete a namespace (and all stacks within)
ocifbsd namespace rm production --force

# Set default namespace for commands
ocifbsd namespace use production
ocifbsd stack deploy -f myapp.yml   # Deploys to production namespace
```

## 8.3 Health Monitoring

### 8.3.1 Health Check Types

Three levels of health monitoring, each serving a different purpose:

| Check Type | Purpose | Failure Action |
|-----------|---------|---------------|
| **Liveness** | Is the pod running? | Restart the pod |
| **Readiness** | Is the pod ready to serve? | Remove from load balancer |
| **Startup** | Has the pod finished starting? | Disable liveness checks during startup |

### 8.3.2 Health Check Configuration

```yaml
pods:
  web:
    replicas: 3
    image: nginx-freebsd:1.25
    healthcheck:
      # Liveness — restart if failing
      liveness:
        cmd: ["/usr/local/bin/curl", "-f", "http://localhost:8080/health"]
        interval: 30s
        timeout: 5s
        retries: 3
        start_period: 60s

      # Readiness — remove from LB if failing
      readiness:
        cmd: ["/usr/local/bin/curl", "-f", "http://localhost:8080/ready"]
        interval: 10s
        timeout: 3s
        retries: 2

      # Startup — wait for initialization
      startup:
        cmd: ["/usr/local/bin/curl", "-f", "http://localhost:8080/started"]
        interval: 5s
        timeout: 3s
        retries: 12   # 60 seconds max
```

### 8.3.3 Health Check Methods

| Method | Description | Example |
|--------|-------------|---------|
| `cmd` | Execute command inside pod | `cmd: ["curl", "-f", "http://localhost/health"]` |
| `http` | HTTP GET request | `http: { path: /health, port: 8080 }` |
| `tcp` | TCP connect | `tcp: { port: 5432 }` |
| `exec` | External command on host | `exec: ["/usr/local/bin/check-pg", "192.168.100.10"]` |

### 8.3.4 Health Monitoring Daemon

The `ocifbsd-healthd` daemon runs on each host and:

1. **Executes checks** at configured intervals
2. **Tracks state transitions** (healthy → unhealthy → healthy)
3. **Notifies the orchestrator** of state changes
4. **Logs all checks** for audit and debugging
5. **Exposes metrics** for external monitoring systems

```
┌─────────────────────────────────────────┐
│         ocifbsd-healthd                  │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │ Check 1 │ │ Check 2 │ │ Check N │   │
│  │ 30s     │ │ 10s     │ │ 5s      │   │
│  └────┬────┘ └────┬────┘ └────┬────┘   │
│       └─────────────┼─────────────┘      │
│                     ▼                    │
│            ┌─────────────┐              │
│            │ State Engine │              │
│            │ (transitions│              │
│            │  & actions) │              │
│            └──────┬──────┘              │
│                   │                     │
│       ┌───────────┼───────────┐        │
│       ▼           ▼           ▼        │
│  ┌────────┐ ┌────────┐ ┌────────┐       │
│  │Restart │ │ Remove │ │  Log   │       │
│  │  Pod   │ │ from LB│ │ Event  │       │
│  └────────┘ └────────┘ └────────┘       │
└─────────────────────────────────────────┘
```

### 8.3.5 Health State Machine

```
         ┌─────────────┐
         │   UNKNOWN   │
         └──────┬──────┘
                │ startup check passes
                ▼
         ┌─────────────┐
         │  STARTING   │◄────────────────┐
         └──────┬──────┘                 │
                │ startup check passes   │ startup check fails
                ▼                      │ (retry)
         ┌─────────────┐               │
    ┌───►│   HEALTHY   │◄──────────────┘
    │    └──────┬──────┘
    │           │ liveness fails
    │           ▼
    │    ┌─────────────┐
    │    │  UNHEALTHY  │
    │    └──────┬──────┘
    │           │ restart
    │           ▼
    │    ┌─────────────┐
    │    │  RESTARTING │
    │    └──────┬──────┘
    │           │ restart completes
    └───────────┘
```

## 8.4 Resource Monitoring

### 8.4.1 Resource Metrics Collection

The `ocifbsd-metricsd` daemon collects resource usage at multiple levels:

| Level | Metrics | Source |
|-------|---------|--------|
| **Host** | CPU, memory, disk I/O, network I/O, load | `sysctl`, `vmstat`, `iostat` |
| **Namespace** | Aggregated pod metrics | Sum of pod metrics |
| **Stack** | Aggregated pod metrics | Sum of pod metrics |
| **Pod** | CPU, memory, disk I/O, network I/O, PIDs | `rctl`, `jail` statistics |
| **Service** | Request rate, latency, error rate | Load balancer statistics |

### 8.4.2 Host Resource Monitoring

```yaml
# Host resource thresholds for automatic actions
host:
  resources:
    memory:
      warning: 75%      # Log warning
      critical: 90%     # Trigger migration
      action: migrate    # Action at critical threshold
    cpu:
      warning: 70%
      critical: 85%
      action: migrate
    disk:
      warning: 80%
      critical: 95%
      action: alert
    load:
      warning: 4.0
      critical: 8.0
      action: migrate

  # Migration policy when host is under pressure
  migration:
    enabled: true
    threshold: critical
    strategy: least-disruption   # or: aggressive, conservative
    max_concurrent: 2            # Max pods migrating at once
    cooldown: 5m                 # Wait between migration rounds
```

### 8.4.3 Pod Resource Monitoring

```yaml
pods:
  web:
    replicas: 3
    image: nginx-freebsd:1.25
    resources:
      memory:
        limit: 512M
        reservation: 256M
      cpus:
        limit: 1.0
        reservation: 0.5

    # Automatic scaling based on resource usage
    autoscale:
      enabled: true
      min_replicas: 2
      max_replicas: 10
      metrics:
        - type: cpu
          target_average: 70%
        - type: memory
          target_average: 80%
        - type: requests_per_second
          target_average: 1000
      scale_up:
        stabilization: 30s
        cooldown: 60s
      scale_down:
        stabilization: 5m
        cooldown: 10m
```

### 8.4.4 Resource Metrics CLI

```bash
# Show host resource usage

# Show namespace resource usage
ocifbsd namespace stats production

# Show stack resource usage
ocifbsd stack stats myapp

# Show pod resource usage

# Show top consumers

# Show resource history
ocifbsd pod stats myapp-web-1 --history 1h
```

## 8.5 Live Migration

### 8.5.1 Migration Architecture

Live migration moves a running pod from one host to another with minimal disruption. The process leverages FreeBSD's jail migration capabilities combined with ZFS send/recv for state transfer.

```
Source Host                    Destination Host
┌─────────────────┐           ┌─────────────────┐
│  Pod: web-1     │           │                 │
│  ┌───────────┐  │           │  ┌───────────┐  │
│  │  Jail     │  │  Step 1  │  │  Jail     │  │
│  │  (running)│  │ ───────► │  │  (created)│  │
│  └───────────┘  │  Create  │  └───────────┘  │
│        │        │  target  │        │        │
│        ▼        │  jail    │        ▼        │
│  ┌───────────┐  │           │  ┌───────────┐  │
│  │  ZFS      │  │  Step 2  │  │  ZFS      │  │
│  │  Dataset  │  │ ───────► │  │  Dataset  │  │
│  │  (live)   │  │  Send    │  │  (syncing)│  │
│  └───────────┘  │  snapshot│  └───────────┘  │
│        │        │           │        │        │
│        ▼        │  Step 3  │        ▼        │
│  ┌───────────┐  │ ───────► │  ┌───────────┐  │
│  │  Network  │  │  Sync    │  │  Network  │  │
│  │  (VNET)   │  │  VNET    │  │  (VNET)   │  │
│  └───────────┘  │  config  │  └───────────┘  │
│        │        │           │        │        │
│        ▼        │  Step 4  │        ▼        │
│  ┌───────────┐  │ ───────► │  ┌───────────┐  │
│  │  Drain    │  │  Stop    │  │  Start    │  │
│  │  (stop    │  │  source, │  │  (accept  │  │
│  │   accepting│  │  start   │  │   traffic)│  │
│  │   traffic)│  │  target  │  │           │  │
│  └───────────┘  │           │  └───────────┘  │
│        │        │  Step 5  │        │        │
│        ▼        │ ───────► │        ▼        │
│  ┌───────────┐  │  Final   │  ┌───────────┐  │
│  │  Destroy  │  │  sync,   │  │  Running  │  │
│  │  (cleanup)│  │  cleanup │  │  (live)   │  │
│  └───────────┘  │           │  └───────────┘  │
└─────────────────┘           └─────────────────┘
```

### 8.5.2 Migration Types

| Type | Downtime | Use Case | Method |
|------|----------|----------|--------|
| **Live** | < 1 second | Stateful services, databases | CRIU-like checkpoint/restore + ZFS sync |
| **Warm** | < 5 seconds | Most applications | Stop, sync, start |
| **Cold** | < 30 seconds | Batch jobs, stateless | Stop, transfer, start |

### 8.5.3 Live Migration Process

```yaml
migration:
  # Migration policy for this stack
  enabled: true
  type: live           # live, warm, cold

  # Pre-migration checks
  pre_check:
    - health_status: healthy
    - disk_usage: < 10G
    - memory_usage: < 80%

  # Connection draining
  drain:
    enabled: true
    timeout: 30s         # Max time to wait for connections
    grace_period: 10s    # Time to finish active requests

  # State transfer
  state_transfer:
    method: zfs_send     # zfs_send, rsync, snapshot
    compression: lz4
    bandwidth_limit: 100M  # Max transfer bandwidth

  # Post-migration verification
  post_check:
    - health_status: healthy
    - service_responsive: true
    - duration: < 10s
```

### 8.5.4 Connection Draining

Before migrating, active connections are gracefully drained:

1. **Signal readiness failure** — Remove pod from load balancer
2. **Wait for active connections** — Allow existing connections to complete
3. **Force close after timeout** — Close remaining connections
4. **Migrate** — Transfer state to new host

```
Client ──► LB ──► Pod (web-1 on host-a)
                    │
                    │ 1. Mark unhealthy
                    ▼
              ┌───────────┐
              │  Draining │  ◄── No new connections
              │  (10s)    │  ◄── Active connections complete
              └───────────┘
                    │
                    │ 2. Connections drained
                    ▼
              ┌───────────┐
              │  Migrate  │  ◄── Transfer to host-b
              │  (5s)     │
              └───────────┘
                    │
                    │ 3. Start on host-b
                    ▼
Client ──► LB ──► Pod (web-1 on host-b)
```

### 8.5.5 Automatic Migration Triggers

Migration is triggered automatically by resource pressure or administrative action:

| Trigger | Condition | Action |
|---------|-----------|--------|
| **Host memory pressure** | Host memory > 90% | Migrate largest memory consumer |
| **Host CPU pressure** | Host CPU > 85% for 5m | Migrate highest CPU consumer |
| **Host disk pressure** | Host disk > 95% | Migrate pods with largest datasets |
| **Host maintenance** | Admin schedules maintenance | Migrate all pods gracefully |
| **Pod resource limit** | Pod exceeds memory limit | Migrate to host with more capacity |
| **Affinity violation** | Pod on wrong host | Migrate to correct host |

### 8.5.6 Migration CLI

```bash
# Migrate a specific pod to another host
ocifbsd pod migrate myapp-web-1 --to host-b

# Migrate with specific strategy
ocifbsd pod migrate myapp-web-1 --to host-b --strategy live

# Migrate all pods from a host (maintenance mode)
ocifbsd host drain host-a

# Migrate a stack to balance load
ocifbsd stack migrate myapp --strategy balanced

# Show migration status
ocifbsd migration ls

# Cancel a migration
ocifbsd migration cancel <migration-id>

# Show migration history
ocifbsd migration history --pod myapp-web-1
```

## 8.6 Live Load Balancing

### 8.6.1 Continuous Load Balancing

Unlike static placement, live load balancing continuously redistributes pods across hosts based on current resource usage and constraints.

```
Time T0:                    Time T1 (after load change):
┌─────────┐ ┌─────────┐    ┌─────────┐ ┌─────────┐
│ Host A  │ │ Host B  │    │ Host A  │ │ Host B  │
│ CPU: 30%│ │ CPU: 40%│    │ CPU: 75%│ │ CPU: 35%│
│         │ │         │    │         │ │         │
│ [web-1] │ │ [web-2] │    │ [web-1] │ │ [web-2] │
│ [api-1] │ │ [api-2] │    │ [api-1] │ │ [api-2] │
│ [db-1]  │ │         │    │         │ │ [db-1]  │◄── Migrated
└─────────┘ └─────────┘    └─────────┘ └─────────┘
```

### 8.6.2 Load Balancing Strategies

| Strategy | Description | Use Case |
|----------|-------------|----------|
| **Balanced** | Equalize CPU/memory across hosts | General purpose |
| **Packed** | Fill hosts completely before using next | Minimize host count |
| **Spread** | Distribute pods as widely as possible | High availability |
| **Bin-packing** | Optimize for resource efficiency | Cost optimization |
| **Affinity-aware** | Respect pod affinity/anti-affinity | Stateful services |

### 8.6.3 Load Balancing Configuration

```yaml
stack: myapp

# Global load balancing settings
load_balance:
  enabled: true
  strategy: balanced
  interval: 5m           # Re-evaluate every 5 minutes
  threshold: 15%         # Trigger when imbalance > 15%

  # Constraints
  constraints:
    max_migrations_per_round: 2
    cooldown: 10m          # Wait between migrations
    business_hours_only: false

  # Pod-specific rules
  pods:
    web:
      strategy: spread     # Spread web servers across hosts
    db:
      strategy: packed     # Keep database pods together
      anti_affinity:
        - pod: db          # But not on same host as other db pods
          max_per_host: 1
```

### 8.6.4 Zero-Downtime Rebalancing

Rebalancing pods without service disruption:

1. **Scale up** — Start new pod on target host
2. **Health check** — Verify new pod is healthy
3. **Add to LB** — Include new pod in load balancer
4. **Drain old pod** — Remove old pod from LB, wait for connections
5. **Scale down** — Stop old pod

```
Step 1: Scale up          Step 2: Health check
┌─────────┐ ┌─────────┐   ┌─────────┐ ┌─────────┐
│ Host A  │ │ Host B  │   │ Host A  │ │ Host B  │
│ [web-1] │ │ [web-2] │   │ [web-1] │ │ [web-2] │
│ [api-1] │ │ [api-2] │   │ [api-1] │ │ [api-2] │
│         │ │         │   │         │ │ [web-3] │◄── New
└─────────┘ └─────────┘   └─────────┘ └─────────┘

Step 3: Add to LB         Step 4: Drain old
┌─────────┐ ┌─────────┐   ┌─────────┐ ┌─────────┐
│ Host A  │ │ Host B  │   │ Host A  │ │ Host B  │
│ [web-1] │ │ [web-2] │   │ [web-1] │ │ [web-2] │
│ [api-1] │ │ [api-2] │   │ [api-1] │ │ [api-2] │
│         │ │ [web-3] │◄──│         │ │ [web-3] │
└─────────┘ └─────────┘   └─────────┘ └─────────┘
                          [web-1 draining...]

Step 5: Scale down
┌─────────┐ ┌─────────┐
│ Host A  │ │ Host B  │
│         │ │ [web-2] │
│ [api-1] │ │ [api-2] │
│         │ │ [web-3] │◄── Now primary
└─────────┘ └─────────┘
```

### 8.6.5 Affinity and Anti-Affinity

Control pod placement for high availability and performance:

```yaml
pods:
  web:
    replicas: 3
    affinity:
      # Prefer to be on hosts with the 'web' label
      node_affinity:
        preferred:
          - match:
              labels:
                role: web
            weight: 100

    anti_affinity:
      # Never put two web pods on the same host
      pod_anti_affinity:
        required:
          - match:
              pod: web
            max_per_host: 1

  db:
    replicas: 2
    affinity:
      # Must be on hosts with SSD storage
      node_affinity:
        required:
          - match:
              labels:
                storage: ssd

    anti_affinity:
      # Spread across availability zones
      pod_anti_affinity:
        required:
          - match:
              pod: db
            max_per_zone: 1
```

### 8.6.6 Load Balancing CLI

```bash
# Show current load distribution
ocifbsd loadbalance status

# Trigger manual rebalancing
ocifbsd loadbalance rebalance --stack myapp --strategy balanced

# Set load balancing strategy for a stack
ocifbsd stack set myapp --load-balance-strategy spread

# Show migration queue
ocifbsd loadbalance queue

# Pause automatic load balancing
ocifbsd loadbalance pause

# Resume automatic load balancing
ocifbsd loadbalance resume
```

## 8.7 Resource Pressure Handling

### 8.7.1 Resource Pressure Levels

| Level | Memory | CPU | Disk | Action |
|-------|--------|-----|------|--------|
| **Normal** | < 75% | < 70% | < 80% | None |
| **Warning** | 75-90% | 70-85% | 80-95% | Log alert |
| **Critical** | > 90% | > 85% | > 95% | Trigger migration |
| **Emergency** | > 95% | > 95% | > 98% | Kill lowest-priority pods |

### 8.7.2 OOM Handling

When a pod exceeds its memory limit:

1. **RCTL triggers** — `memoryuse` limit exceeded
2. **Notify orchestrator** — Event sent to `ocifbsd-orchestrator`
3. **Decision:**
   - If pod has `restart: always` → Restart with increased limit (if configured)
   - If host has capacity → Migrate to another host
   - If no options → Kill and alert

```yaml
pods:
  web:
    resources:
      memory:
        limit: 512M
        oom_policy:
          action: restart      # restart, migrate, kill
          increase_limit: true
          increase_by: 128M
          max_limit: 2G
```

### 8.7.3 CPU Throttling

When a pod exceeds its CPU limit:

1. **RCTL throttles** — `pcpu` limit enforced
2. **Metrics show throttling** — `ocifbsd_pod_cpu_throttled_seconds`
3. **Auto-scale** — If throttling persists, scale up replicas
4. **Alert** — Notify if throttling exceeds threshold

## 8.10 Conclusion

The Namespace and Resource Management system provides:

- **True multi-tenancy** — Namespaces isolate resources, networks, and security domains
- **Proactive health monitoring** — Detect and respond to issues before they cause failures
- **Intelligent resource management** — Monitor, alert, and act on resource pressure automatically
- **Graceful live migration** — Move pods between hosts with minimal disruption
- **Continuous load balancing** — Keep workloads optimally distributed across the cluster

This system ensures that FreeBSD native containers run reliably at scale, with automatic recovery from resource pressure and host failures.
