> **Purpose:** This document defines how autonomous agents (and human developers) should consume the plan documents, pick tasks, claim work, commit changes, and handle merge conflicts when multiple agents work on the same branch concurrently.
>
> **Note for agents:** Read this document **before** starting any task. Follow these instructions exactly. If any step is unclear, ask for clarification.

---

## 0.1.1 Overview

This project uses a **shared branch model** where multiple agents may be working on the same Git branch simultaneously. To avoid conflicts and ensure traceability, every task must be claimed, tracked, and completed following a strict protocol.

**Key principles:**
- **Claim before work** — Never start a task without first claiming it in the document
- **Check dependencies** — Never start a task whose dependencies are not yet complete
- **Commit frequently** — Commit after each task completion, not after every file edit
- **Handle conflicts gracefully** — If a merge conflict occurs, determine if the task was taken by another agent

## 0.1.2 Task Selection

### 0.1.2.1 How to Pick a Task

1. **Consult the TOC** — Open [`0.0-OCI-Jail-TOC.md`](./0.0-OCI-Jail-TOC.md) to understand the document structure
2. **Read the relevant document** — Open the document containing the task you want to work on
3. **Find an unclaimed task** — Look in the task tables for a row where:
   - `Status` is empty or `⬜ PENDING`
   - `Assigned To` is empty
   - `Start` is empty
4. **Check dependencies** — Verify that all tasks listed in the `Dependencies` column are marked as `✅ DONE` or have a completed `End` date
5. **If dependencies are not met** — Do NOT start the task. Either:
   - Pick a different task whose dependencies are met, OR
   - Work on completing the blocking dependency first

### 0.1.2.2 Priority Order

Tasks should generally be worked on in the following order:

1. **Phase 0 tasks** (foundation and setup) — highest priority, blocks everything
2. **Phase 1 tasks** (OCI runtime core) — blocks all container operations
3. **Phase 2 tasks** (image management) — blocks networking and security
4. **Phase 3 tasks** (networking) — blocks resource limits and security
5. **Phase 4 tasks** (resource limits and security) — blocks integration
6. **Phase 5 tasks** (integration and polish) — quality assurance
7. **Phase 6 tasks** (orchestration) — feature implementation
8. **Phase 7 tasks** (config conversion) — feature implementation

Within each phase, tasks with the most dependencies should be started first.

## 0.1.3 Claiming a Task

### 0.1.3.1 Before You Start

Before claiming a task, ensure:

- [ ] You have read this workflow document
- [ ] You have read the relevant plan document(s)
- [ ] You understand the task requirements
- [ ] All dependencies are marked as complete

### 0.1.3.2 How to Claim

1. **Open the document** containing the task (e.g., `5.0-Risks-TODO.md`)
2. **Find the task row** in the table
3. **Set `Status`** to `🔄 IN PROGRESS`
4. **Set `Assigned To`** to your agent identifier (e.g., `Junie-Agent-1`)
5. **Set `Start`** to the current date/time in ISO 8601 format: `YYYY-MM-DD HH:MM UTC`
6. **Save the document**

**Example — Before claiming:**

| ID | Task | Status | Assigned To | Start | End | Dependencies | Files | Notes |
|----|------|--------|-------------|-------|-----|--------------|-------|-------|
| 1.1 | Create `usr.sbin/ocifbsd/` directory structure | ⬜ PENDING | | | | 0.1 | | Makefile, headers, subdirs |

**Example — After claiming:**

| ID | Task | Status | Assigned To | Start | End | Dependencies | Files | Notes |
|----|------|--------|-------------|-------|-----|--------------|-------|-------|
| 1.1 | Create `usr.sbin/ocifbsd/` directory structure | 🔄 IN PROGRESS | Junie-Agent-1 | 2026-04-28 22:00 UTC | | 0.1 | | Makefile, headers, subdirs |

### 0.1.3.3 Commit and Push After Claiming

After claiming a task, immediately commit and push the change:

```bash
git add .plan/<document>.md
git commit -m "Claim task <ID>: <brief description>" --trailer "Co-authored-by: Junie <junie@jetbrains.com>"
git push
```

**Important:** You must push immediately after committing. Other agents working on the same branch need to see your claim to avoid starting the same task. The claim is not effective until it is pushed to the remote repository.

## 0.1.4 Working on a Task

### 0.1.4.1 Implementation Guidelines

1. **Follow the plan** — Implement exactly what the task describes. Do not add scope.
2. **Follow code style** — Match the existing codebase conventions (see `CONTRIBUTING.md` if present)
3. **Write tests** — Every task that involves code changes must include tests
4. **Document changes** — Update relevant `.plan` documents if the implementation differs from the plan

### 0.1.4.2 If You Get Stuck

If you encounter a blocker that prevents task completion:

1. **Update `Notes`** in the task table with the blocker description
2. **Set `Status`** to `⏸️ BLOCKED`
3. **Commit and push** the status update
4. **Pick a different task** whose dependencies are met

---

## 0.1.5 Completing a Task

### 0.1.5.1 How to Mark Complete

When a task is fully implemented and verified:

1. **Run all unit tests** — Execute the full test suite for the affected module(s). Do NOT consider a task complete unless **all** unit tests pass.
2. **Fix other agents' code if needed** — If tests fail due to issues in another agent's code (e.g., a dependency task has bugs), fix those issues as part of completing your task. Do not leave broken tests for someone else.
3. **Set `Status`** to `✅ DONE`
4. **Set `End`** to the current date/time in ISO 8601 format: `YYYY-MM-DD HH:MM UTC`
5. **Update `Notes`** — Add a brief summary of what was implemented (1-2 sentences), including any fixes made to other agents' code
6. **Update `Files`** — Ensure all files created/modified are listed, including files modified in other agents' tasks

**Example — After completion:**

| ID | Task | Status | Assigned To | Start | End | Dependencies | Files | Notes |
|----|------|--------|-------------|-------|-----|--------------|-------|-------|
| 1.1 | Create `usr.sbin/ocifbsd/` directory structure | ✅ DONE | Junie-Agent-1 | 2026-04-28 22:00 UTC | 2026-04-28 23:15 UTC | 0.1 | `usr.sbin/ocifbsd/Makefile`, `usr.sbin/ocifbsd/ocifbsd.h` | Created directory structure with Makefile and common header |

### 0.1.5.2 Commit and Push After Completion

After marking a task complete, commit and push:

```bash
git add -A
git commit -m "Complete task <ID>: <brief description>" --trailer "Co-authored-by: Junie <junie@jetbrains.com>"
git push
```

**Important:** You must push immediately after committing. Other agents working on the same branch need to see your completed work. The task completion is not effective until it is pushed to the remote repository.

## 0.1.6 Handling Merge Conflicts

### 0.1.6.1 Before You Start Work

Always pull the latest changes before starting work:

```bash
git pull --rebase
```

### 0.1.6.2 If a Merge Conflict Occurs

If `git pull --rebase` results in a merge conflict:

1. **Identify the conflicting files**
2. **Determine if the conflict is in a task table** — If yes, check if another agent has claimed or completed the same task
3. **If another agent has claimed the task** — Abandon your changes for that task and pick a different one
4. **If another agent has completed the task** — Use their version (the completed work)
5. **If the conflict is in code** — Resolve manually, preferring the more complete implementation
6. **If unsure** — Ask for clarification before proceeding

### 0.1.6.3 Conflict Resolution Priority

When resolving conflicts, prefer in this order:

1. **Completed tasks** over in-progress tasks
2. **In-progress tasks** over pending tasks
3. **Code that passes tests** over code that does not
4. **The first agent to claim** (earlier timestamp) if both are in progress

## 0.1.7 Communication

### 0.1.7.1 Commit Messages

Use clear, descriptive commit messages:

- **Claiming:** `Claim task <ID>: <brief description>`
- **Completing:** `Complete task <ID>: <brief description>`
- **Updating:** `Update task <ID>: <what changed>`
- **Fixing:** `Fix task <ID>: <what was fixed>`

### 0.1.7.2 Notes Field

Use the `Notes` field in task tables for:

- Blockers or dependencies discovered during work
- Design decisions made during implementation
- Links to related commits or PRs
- Known issues or TODOs for future work
